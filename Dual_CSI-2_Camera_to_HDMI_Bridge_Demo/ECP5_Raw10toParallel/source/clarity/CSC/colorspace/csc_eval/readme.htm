CSC Generator IP Core

Implementing the IP Module Using Diamond


Instantiating the Core

The generated CSC Generator IP core package includes black-box (<user_name>_bb.v) and instance (<username>_inst.v) templates that can be used to instantiate the core in a top-level design.

An example RTL top-level reference source file that can be used as an instantiation template for the IP core is provided in <project_dir>\csc_eval\<username>\src\rtl\top\<family>. Users may also use this top-level reference as the starting template for the top-level for their complete design.

For VHDL entry, only a top-level file is generated which contains the core instantiation within.

For Linux/Unix environment:
Once the IP core is installed, the user needs to change the permissions of the files in the install directory. To do this, go to <install_directory>/csc_core_v3.0/gui and type chmod 777 *_l(s). Otherwise, when the user tries to generate the IP core, IPexpress will error out.


Hardware Evaluation

Lattice's IP hardware evaluation capability makes it possible to create IP cores that operate in hardware for a limited period of time (approximately four hours) without requiring the purchase on an IP license. The hardware evaluation capability is enabled by default. It can be disabled by right clicking on "Build Database" in the "Process for current sources" window of the Project Navigator. The setting is called "Hardware Evaluation" and the options are "Enable" or "Disable". If Diamond tool is used, hardware evaluation capability can be enabled or disabled by clicking on “Translate design” in Strategies window and set the option “Hardware Evaluation”.
 
When the Hardware Evaluation feature is enabled in the design, it will generate a programming file that may be downloaded into the device. After initialization, the IP core will be operational for approximately four hours. After four hours, the device will stop working and it will be necessary to reprogram the device to re-enable operation. This hardware evaluation capability is only enabled if the core has not been licensed. During implementation, a license check is performed. If the hardware evaluation feature is disabled, a pop-up window will be displayed indicating a license failure. Click"OK" in the window and the bitstream will not be generated. If a license is detected, no pop-up window is displayed and core generation is completed with no restrictions.

 


Implementing the Core in a Top-Level Design


As described previously, the top-level file <username>_top.v provided in <project_dir>\csc_eval\<username>\src\rtl\top\<family> supports the ability to implement just the CSC_Generator Block.

Push-button top-level implementation of this top-level is supported via Diamond project file <username>_top.ldf located in <project_dir>\csc_eval\<username>\impl\<synthesis>.

This design is intended only to provide an accurate indication of the device utilization associated with the core itself and should not be used as an actual implementation example.


To use the project file:

  • Select Open Project (Open -> Project) under the File tab in Diamond.
  • Browse to <project_dir>\csc_eval\<username>\impl\<synthesis> in the Open Project dialog box.
  • Select and open <username>_top.ldf . At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  • One lpf file is available in the \impl\<synthesis> directory <username>_top.lpf.
  • Implement the complete design via the standard Diamond GUI flow.
  • All other options remain at their default values.


Running Functional Simulation

The functional simulation includes a configuration-specific behavioral model of the CSC Generator, which is instantiated in an FPGA top level along with some other logic (like PLLs and registers with Read/Write Interface). This FPGA top is instantiated in an evaluation testbench that provides appropriate stimulus for the CSC Generator IP core. The testbench files can be found in <project_dir>\csc_eval\testbench.

 

The generated IP core package includes the configuration-specific behavior model (<username>_beh.v,provided in <project_dir>\csc_eval\<username>\src\beh_rtl\<family>) for functional simulation.Models for simulation are provided in the corresponding \models folder if required.
Users may run the eval simulation on ModelSim SE* by doing the following:

  • Open ModelSim SE.
  • Under the Tools tab, select TCL -> Execute Macro…
  • Select file <project_dir>\csc_eval\<username>\sim\modelsim\scripts\<username>_rtl_se.do
  • Select OK.

 

Users may run the eval simulation on Active-HDL(Windows only) by doing the following:

  • Open Active-HDL.
  • Under the Tools tab, select Execute Macro...
  • Select file <project_dir>\csc_eval\<username>\sim\aldec\scripts\<username>_rtl.do
  • Select OK.

Running Timing Simulation

In order to run the Post route timing simulation, an IP license is required. You need to generate the timing simulation files by double-clicking on Generate Timing Simulation Files in the process for current source window of the Project Navigator.

Users may run the eval simulation on ModelSim SE(1) by doing the following:

  • Open ModelSim SE.
  • Under the Tools tab, select TCL -> Execute Macro…
  • Select file <project_dir>\csc_eval\<username>\sim\modelsim\scripts\<username>_postroute_synplify(lse)_se.do
  • Select OK.

 

Users may run the eval simulation on Active-HDL(Windows only) by doing the following:

  • Open Active-HDL.
  • Under the Tools tab, select Execute Macro...
  • Select file <project_dir>\csc_eval\<username>\sim\aldec\scripts\<username>_postroute_synplify(lse).do
  • Select OK.

 

(1)    For running VHDL RTL/Timing simulation on ModelSim SE, pre-compiled Library should be prepared first.
