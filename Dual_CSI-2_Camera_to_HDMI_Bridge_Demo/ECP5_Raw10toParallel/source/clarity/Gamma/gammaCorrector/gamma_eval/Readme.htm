Gamma Corrector IP Core

Implementing the IP Module Using Lattice Diamond


Instantiating the Core

The generated Gamma Corrector IP core package contains Verilog and VHDL files. The Verilog HDL files include black-box (<username>_bb.v) and instance (<username>_inst.v) templates that can be used to instantiate the core in a top-level design.

An example RTL top-level reference source file that can be used as an instantiation template for the IP core is provided in <project_dir>\gamma_eval\<username>\src\rtl\top\<family>. Users may also use this top-level reference as the starting template for the top-level for their complete design.

For VHDL entry, only a top-level file is generated which contains the core instantiation within.


Hardware Evaluation

Lattice's IP hardware evaluation capability makes it possible to create IP cores that operate in hardware for a limited period of time (approximately four hours) without requiring the purchase on an IP license. The hardware evaluation capability is enabled by default. It can be disabled by Project->Active Strategy->Translate Design Settings. The setting is called "Hardware Evaluation" and the options are "Enable" or "Disable".
 
When the Hardware Evaluation feature is enabled in the design, it will generate a programming file that may be downloaded into the device. After initialization, the IP core will be operational for approximately four hours. After four hours, the device will stop working and it will be necessary to reprogram the device to re-enable operation. This hardware evaluation capability is only enabled if the core has not been licensed. During implementation, a license check is performed. If the hardware evaluation feature is disabled, a pop-up window will be displayed indicating a license failure. Click"OK" in the window and the bitstream will not be generated. If a license is detected, no pop-up window is displayed and core generation is completed with no restrictions.


Implementing the Core in a Top-Level Design


As described previously, the top-level file <username>_top.v(vhd) provided in <project_dir>\gamma_eval\<username>\src\rtl\top\<family> supports the ability to implement just the Gamma Corrector Block.

Push-button implementation of this top-level is supported via the Diamond project file <username>_top.ldf located in <project_dir>\gamma_eval\<username>\impl\<synplify|LSE>.

This design is intended to provide an indication of the device utilization associated with the core itself and is usually replaced with the user's top-level design.

 

To use the project file:

  • Select Open -> Project under the File tab in Diamond.
  • Browse to <project_dir>\gamma_eval\<username>\impl\<synplify|LSE>in the Open Project dialog box.
  • Select and open <username>. ldf. At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  • One lpf file is available in the \impl\<synplify|LSE> directory <username>_top.lpf.
  • Implement the complete design via the standard Diamond GUI flow, keeping all other options at their default values.

Running Functional Simulation

The functional simulation includes a configuration-specific behavioral model of the Gamma Corrector, which is instantiated in an FPGA top level along with some other logic (like PLLs and registers with Read/Write Interface). This FPGA top is instantiated in an evaluation testbench that provides appropriate stimulus for the Gamma Corrector IP core. The testbench files can be found in <project_dir>\gamma_eval\testbench.

 

The generated IP core package includes the configuration-specific behavior model (<username>_beh.v,provided in <project_dir>\gamma_eval\<username>\src\beh_rtl\<family>) for functional simulation.Models for simulation are provided in the corresponding \models folder if required.

 

Users may run the eval simulation on ModelSim SE*(Windows or Linux) by doing the following:

  • Open ModelSim SE.
  • Under the Tools tab, select TCL -> Execute Macro…
  • Select file <project_dir>\gamma_eval\<username>\sim\modelsim\scripts\<username>_rtl.do.
  • Select OK.

 

Users may run the eval simulation on Active-HDL(Windows only) by doing the following:

  • Open Active-HDL.
  • Under the Tools tab, select Execute Macro...
  • Select file <project_dir>\gamma_eval\<username>\sim\aldec\scripts\<username>_rtl.do
  • Select OK.

Running Timing Simulation

In order to run the Post route timing simulation, an IP license is required. You need to generate the timing simulation files by double-clicking on “Generate Timing Simulation Files” in the process for current source window of the Project Navigator.

Users may run the eval simulation on ModelSim SE*(Windows or Linux) by doing the following:

  • Open ModelSim SE.
  • Under the Tools tab, select TCL -> Execute Macro..
  • Select file <project_dir>\gamma_eval\<username>\sim\modelsim\scripts\<username>_sdf.do.
  • Select OK.

 

Users may run the eval simulation on Active-HDL(Windows only) by doing the following:

  • Open Active-HDL.
  • Under the Tools tab, select Execute Macro...
  • Select file <project_dir>\gamma_eval\<username>\sim\aldec\scripts\<username>_sdf.do
  • Select OK.

 

* For running VHDL Rtl/Timing simulation on ModelSim SE, pre-compiled Library should be prepared first.
* For some Timing simulation on Modelsim,you may see some error setup-time information within 7 ns, it's not problem as long as the NUM_ERRORs is zero at the end of the test.
* The timing simulation script runs without SDF file by default. If SDF delays are needed, add a PLL to the top-level and regenerate the timing simulation files.

 

Timing Report file

·        Under File tab, select Open -> Project.

·        Select file <project_dir>\gamma_eval\<username>\impl\synplify|LSE\<username>_top.ldf
